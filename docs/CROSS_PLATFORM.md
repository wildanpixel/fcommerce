# Cross-Platform Desktop Notes

## Supported Targets

The same Electron, React, TypeScript, Node, Express, Prisma, SQLite, Playwright, and Puppeteer codebase targets:

- Windows installer: `apps/desktop/release/MarketPlace Keyword Competitor Analysis Setup 1.0.0.exe`
- Windows portable: `apps/desktop/release/MarketPlace Keyword Competitor Analysis Portable 1.0.0.exe`
- macOS app: `MarketPlace Keyword Competitor Analysis.app`, generated by `electron-builder --mac dir`
- macOS DMG: `MarketPlace Keyword Competitor Analysis 1.0.0-arm64.dmg` and `MarketPlace Keyword Competitor Analysis 1.0.0-x64.dmg`, generated by `electron-builder --mac dmg`

Linux is not packaged in V1, but platform services and browser discovery are structured so Linux can be added without changing business logic.

## Application Folders

All runtime data is resolved through `PlatformService`.

Windows uses Electron `app.getPath("userData")`, normally under:

`%APPDATA%/MarketPlace Keyword Competitor Analysis`

macOS uses Electron `app.getPath("userData")`, normally under:

`~/Library/Application Support/MarketPlace Keyword Competitor Analysis`

The app stores:

- Database: `data/marketplace-intelligence.db`
- Settings and encrypted secrets: `settings/`
- Screenshots: `screenshots/`
- Reports: `reports/`
- Logs: `logs/`
- Cache and browser profiles: `cache/` and `browser-profiles/`

## Browser Detection

Browser automation uses `BrowserDiscoveryService` and `PlaywrightBrowserLauncher`.
Platform-specific browser executable locations are exposed through `PlatformService`; collectors do not hardcode operating-system paths.

Supported choices:

- Bundled Chromium
- Google Chrome
- Microsoft Edge
- Brave

Chrome, Edge, and Brave are detected with platform-specific locations and the system `PATH`. No Windows executable path is hardcoded. If the chosen browser is unavailable, the automation falls back to bundled Chromium.

Browser sessions use persistent profiles stored under the platform app folder, one profile per browser.

## Android Tooling

Android support is implemented behind the Android automation adapter boundary and does not change marketplace adapter contracts.

Tool discovery checks:

- `ANDROID_HOME`
- `ANDROID_SDK_ROOT`
- Windows `%LOCALAPPDATA%/Android/Sdk`
- macOS `~/Library/Android/sdk`
- Linux `~/Android/Sdk`
- System `PATH`

Detected tools and runtime state:

- ADB
- Android Emulator CLI
- SDK manager
- AVD manager
- Java runtime
- Android Virtual Devices
- Connected Android devices
- Device boot completion
- TikTok package IDs: `com.zhiliaoapp.musically` and `com.ss.android.ugc.trill`
- TikTok runtime state, active ANR state, focused activity, package ABI, and device ABI
- Local TikTok APK candidates from Downloads/Desktop folders

The TikTok workflow launches a native Android emulator window and keeps collection controls in the Electron app. TikTok APKs are not bundled; users can select a local APK for ADB install or use the emulator's Play Store flow.

Android screenshots are captured with `adb exec-out screencap -p`. Visible text is extracted with UIAutomator XML dumps when a booted device is available.

The local Windows validation profile is `MIO_TikTok_Stable`, created from the Android 35 Google Play x86_64 system image. This lets a user sign in with Gmail and install TikTok from Play Store, or install a trusted local TikTok APK through the app.

Emulator launch preserves state. The app does not pass `-wipe-data`, clear TikTok app data, uninstall TikTok, or reset Google login when the emulator is closed and reopened. If Android reports TikTok is not responding, the Recover TikTok action force-stops and reopens the app without clearing user data. ARM-only TikTok APKs can still hang on x86_64 emulators because they run through translation; Play Store install or a universal/x86-compatible APK is preferred when available.

## Native Features

Electron native capabilities are centralized in the platform layer:

- Open folders and URLs
- Folder picker
- File picker
- Clipboard copy
- Notifications
- Window lifecycle
- Platform-specific keyboard modifier

Menus use `CommandOrControl`, so shortcuts map to `Ctrl` on Windows and `Command` on macOS.

## Build Commands

Install dependencies:

```bash
pnpm install
```

Validate:

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm test:e2e
pnpm build
```

Package Windows:

```bash
pnpm package:win
```

Package macOS on macOS:

```bash
pnpm package:mac
```

Build both on capable CI:

```bash
pnpm package:all
```

## GitHub Actions

`.github/workflows/build.yml` packages each operating system on its native runner:

- Windows runner: installs dependencies, generates Prisma Client, runs checks, builds React/Electron code, packages NSIS installer and portable EXE, then uploads `.exe` artifacts.
- macOS runner: installs dependencies, generates Prisma Client, runs checks, builds React/Electron code, packages `.app` directories and `.dmg` artifacts for x64 and arm64, then uploads macOS artifacts.
- macOS installation and download instructions are maintained in [MACOS_INSTALLATION.md](MACOS_INSTALLATION.md).

Prisma is configured with `native`, `windows`, `darwin`, and `darwin-arm64` binary targets so packaged desktop builds include the correct SQLite query engines.

## Code Signing And Notarization

Code signing is prepared for environment-driven configuration. Do not hardcode certificates.

Windows signing can use Electron Builder variables such as:

- `CSC_LINK`
- `CSC_KEY_PASSWORD`

Apple signing and notarization can use:

- `APPLE_ID`
- `APPLE_APP_SPECIFIC_PASSWORD`
- `APPLE_TEAM_ID`
- `CSC_LINK`
- `CSC_KEY_PASSWORD`

Signing and notarization should be configured in CI or release environment variables, not in application logic.

## Auto Update

`UpdateService` defines the future update abstraction. V1 ships with `NoopUpdateService`; later providers can target GitHub Releases, a private update server, or another update channel without changing marketplace or report logic.
